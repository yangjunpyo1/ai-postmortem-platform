## 2. 배경 및 문제 정의

인프라 구축 및 실습 과정에서 트러블슈팅이 발생할 때마다
원인과 해결 과정을 수동으로 정리해야 했고
이 과정이 번거롭고 비효율적이라는 것을 직접 경험했다.
기업 엔지니어들도 서비스 장애 시
동일한 과정을 반복해야 한다는 점에서
AI를 활용한 자동화의 필요성을 느꼈다.

### 직접 경험한 문제점

- 트러블슈팅 발생 시 로그를 직접 확인하며
  원인을 수동으로 파악하고 정리해야 함
- 매번 형식 없이 수동으로 기록해
  나중에 찾아보기 어려움
- 트러블슈팅 히스토리가 체계적으로 관리되지 않아
  같은 문제가 반복될 위험이 있음

### 기존 방식의 한계

- 트러블슈팅 기록을 노션, 구글 docs 등에
  수동으로 작성하는 방식은
  형식이 없고 시간이 걸리며 누락 위험이 있음
- 체계적인 히스토리 관리가 어려워
  같은 문제가 반복될 가능성이 높음

## 3. 목표

AWS CloudWatch로 장애를 자동 감지하고
장애 종료 시 슬랙 대화를 자동 수집하여
Claude AI가 Postmortem 문서를 자동 생성하고
웹 대시보드에서 팀 전체가 조회/관리할 수 있는
경량 자동화 플랫폼 구축

---

## 4. 타겟 사용자

- MSP 클라우드 엔지니어
- DevOps 엔지니어
- AWS 기반 서비스를 운영하는 중소기업/스타트업 팀
- 별도 툴 없이 수동으로 postmortem을 작성하는 팀

## 5. 핵심 가치

| 기존 방식 | 이 플랫폼 |
|---|---|
| 수동으로 로그 확인 후 정리 | AI가 자동 분석 및 문서 생성 |
| 형식 없이 노션/구글 docs 기록 | 일관된 형식 자동 유지 |
| 개인 보관으로 히스토리 누락 | 자동 누적 관리 |
| 반복 장애 발생 위험 | 히스토리 기반 재발 방지 |

---

## 6. 주요 기능 (MVP)

1. AWS CloudWatch 장애 자동 감지
2. 심각도별 슬랙 알림
   - 🔴 Critical: 즉시 알림 + 온콜 호출
   - 🟡 Warning: 슬랙 알림만
   - 🟢 Info: 채널에 기록만
3. 장애 종료 시 슬랙 대화 자동 수집
4. Claude AI Postmortem 문서 자동 생성
5. Grafana 메트릭 시각화
6. 웹 대시보드 (장애 목록, 문서 조회, 통계)

## 10. 프로젝트 의의

이전 개별 프로젝트에서 Ansible을 활용해
여러 노드의 환경 구성을 코드 한 줄로 자동화하면서
수동 작업 대비 시간과 실수가 크게 줄어드는 것을 직접 경험했다.
이 자동화의 편의성에서 영감을 받아,
이번에는 장애 대응 후 가장 번거로운 작업인
Postmortem 문서 작성을 자동화하고자 했다.
AWS 클라우드 환경과 Claude AI를 결합하여
엔지니어가 장애 해결에만 집중할 수 있도록
사후 처리 과정을 자동화하는 플랫폼을 구축한다.
